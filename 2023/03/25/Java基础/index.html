BIO/NIO/AIO

  • Blocking IO同步阻塞，一个连接一个线程，即客户端有请求是服务端就会启动一个线程来处理，如果这个连接不做任何事情，则会造成额外开销，可以通过线程池来改善。
  • Non-Blocking IO同步非阻塞，客户端发送的连接请求都会注册到多路复用器上，多路复用器轮询到连接有 I/O 请求时才启动一个线程进行处理。
    • JDK 的 NIO 底层由 epoll 实现。
  • Asynchronous-Non-Blocking-IO异步非阻塞，线程发起 IO 请求，立即返回；内存做好 IO 操作的准备之后，做 IO 操作，直到操作完成或者失败，通过调用注册的回调函数通知线程做 IO 操作完成或者失败。

使用场景

  • BIO：连接数比较小且固定，JDK1.4 之前唯一的选择。
  • NIO：连接数多且连接比较短，比如聊天服务器，弹幕系统，服务器间通讯等，JDK1.4开始引入。
  • AIO：连接数多且连接比较长，比如相册服务器，JDK7开始引入。

例子

  • 同步阻塞：你到饭馆点餐，然后在那等着，啥都干不了，餐没做好，你就必须等着！
  • 同步非阻塞：你在饭馆点完餐，就去玩儿了。不过玩一会儿，就回饭馆问一声：好了没？
  • 异步非阻塞：饭馆打电话说，我们知道您的位置，一会给你送过来，安心玩儿就可以了，类似于外卖。

面向对象

  • 对象：对象是人们要研究的任何事物，他不仅能表示具体的事物，也能表示抽象的规则，计划或者事件。对象是数据和操作的结合，使数据和操作封装在统一的对象中。
  • 类；具有相同特性和操作的的对象的抽象就是类。对象的抽象是类，类的具体化就是对象。
  • 三大特性：
    • 封装：让某个类型的对象获得另一个类型的对象的属性的方法。
    • 继承：隐藏部分对象的属性和实现细节，对数据的访问只能通过外公开的接口。
    • 多态：对于同一个行为，不同的子类对象具有不同的表现形式。
    • 多态存在的3个条件：1）继承；2）重写；3）父类引用指向子类对象。

JDK1.8之后有哪些新特性？

  1. Lambda 表达式和函数式接口：Lambda 表达式本质上是一段匿名内部类，也可以是一段可以传递的代码。Lambda 允许把函数作为一个方法的参数（函数作为参数传递到方法中），使用 Lambda 表达式使代码更加简洁，但是也不要滥用，否则会有可读性等问题，
  2. Stream API：用函数式编程方式在集合类上进行复杂操作的工具，配合Lambda表达式可以方便的对集合进行处理。Java8 中处理集合的关键抽象概念，它可以指定你希望对集合进行的操作，可以执行非常复杂的查找、过滤和映射数据等操作。使用Stream API 对集合数据进行操作，就类似于使用 SQL 执行的数据库查询。也可以使用 Stream API 来并行执行操作。简而言之，Stream API 提供了一种高效且易于使用的处理数据的方式。

Java中有哪些容器（集合类）？

  • Set代表无序的，元素不可重复的集合；
  • List代表有序的，元素可以重复的集合；
  • Queue代表先进先出（FIFO）的队列；
  • Map代表具有映射关系（key-value）的集合。
  • Collection 单列集合
    • List 元素可重复，有序
      • ArrayList 底层为数组，查询快，增删慢
      • LinkedList 底层为双端链表，查询慢，增删快
    • Set 元素不可重复，无序
      • HashSet 底层为HashMap，数组加链表实现
      • TreeSet 底层为红黑树，有序
  • Map 双列集合
    • HashMap 底层数组加链表
    • TreeMap 底层为红黑树，有序
    • LinkedHashMap 底层HashMap和LinkedList实现，通过HashMap实现key-value键值对存储，通过链表实现元素有序。

Java中的容器，线程安全和线程不安全的分别有哪些？

  • 线程不安全容器：比如说ArrayList HashMap HashSet TreeSet都不是
  • 线程安全容器：在JUC（java.util.concurrent）下的都是线程安全的，还有vector，hashtable但不建议使用。
    • java.util.concurrent.ConcurrentHashMap：线程安全的哈希表实现；
    • java.util.concurrent.CopyOnWriteArrayList：线程安全的动态数组实现；
  • 需要注意的是，对于线程不安全的容器，可以通过使用synchronized关键字或者java.util.concurrent.locks包中的锁来实现线程安全。

介绍一下HashMap的扩容机制

  • 数组的初始容量为16，而容量是以2的次方扩充的，一是为了提高性能使用足够大的数组，二是为了能使用位运算代替取模预算(据说提升了5~8倍)。默认阈值是0.75.
  • JDK1.8的rehash算法优化就是对原来的链表或者红黑树进行拆分成两部分，然后分别挂在原来数组的位置和数组的位置 + oldCap的位置，这样做就避免了大量的节点进行重新hash寻址算法和重新放到hash表的过程，大大增加了扩容效率。

HashMap 和 ConcurrentHashMap底层实现的差别。

HashMap是线程不安全的，它的底层实现是一个数组，每个数组元素都是一个链表，当出现哈希冲突时，会将新元素插入到链表的头部。当链表长度过长时，HashMap会将链表转换为红黑树，以提高查询效率。
ConcurrentHashMap是线程安全的，并发读写时使用了分段锁，底层实现是一个由多个小的哈希表组成的数组，每个小哈希表称为“段”，每个段都有自己的锁。在ConcurrentHashMap中，只有当一个线程占据了一个段的锁时，才会受到该段上的限制。这样，多个线程可以同时读取和写入不同的段，从而提高并发性能。当需要对ConcurrentHashMap进行扩容时，只需要对某一个段进行扩容，而不是对整个哈希表进行扩容，因此ConcurrentHashMap的扩容效率更高。

ConcurrentHashMap 在 JDK 1.7 时使用的是数据加链表的形式实现的，其中数组分为两类：大数组 Segment 和小数组 HashEntry，而加锁是通过给 Segment 添加 ReentrantLock 锁来实现线程安全的。而 JDK 1.8 中 ConcurrentHashMap 使用的是数组+链表/红黑树的方式实现的，它是通过 CAS 或 synchronized 来实现线程安全的，并且它的锁粒度更小，查询性能也更高。

什么是CopyOnWrite容器

  • CopyOnWrite容器即写时复制的容器。通俗的理解是当我们往一个容器添加元素的时候，不直接往当前容器添加，而是先将当前容器进行Copy，复制出一个新的容器，然后新的容器里添加元素，添加完元素之后，再将原容器的引用指向新的容器。
  • 读的时候不需要加锁，如果读的时候有多个线程正在向CopyOnWriteArrayList添加数据，读还是会读到旧的数据，因为开始读的那一刻已经确定了读的对象是旧对象。写是加锁 lock。
  • CopyOnWrite并发容器用于读多写少的并发场景。比如白名单，黑名单等场景。
  • 内存占用问题 数据一致性问题 CopyOnWriteArrayList读取时不加锁，只是写入、删除、修改时加锁

Java的内存泄漏和内存溢出

  • 内存泄露是指程序在申请内存后，无法释放已申请的内存空间。
  • 内存溢出是指程序所需要的内存超出了系统所能分配的内存（包括动态扩展）的上限。
    • Java堆溢出：不断的创建对象，并保证GC Roots到对象之间有可达路径
    • Java栈溢出：请求的栈深度大于虚拟机锁允许的最大深度
    • 方法区和运行时常量池溢出
