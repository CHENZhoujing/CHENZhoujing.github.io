分布式
Mar 29 2023

CAP原则

  • 一致性 “all nodes see the same data at the same time”
  • 可用性 “Reads and writes always succeed”
  • 分区容错性 即分布式系统在遇到某节点或网络分区故障的时候，仍然能够对外提供满足一致性和可用性的服务
  • 最多只能同时三个特性中的两个，三者不可兼得。

高并发

  • 高并发意味着大流量
  • 我们常见的高并发场景有：淘宝的双11、春运时的抢票、微博大V的热点新闻等。除了这些典型事情，每秒几十万请求的秒杀系统、每天千万级的订单系统、每天亿级日活的信息流系统等，都可以归为高并发。
  • 高并发系统的目的：
    • 高性能 性能体现了系统的并行处理能力，在有限的硬件投入下，提高性能意味着节省成本。
      • 平均响应时间
      • TP90、TP99等分位值
    • 高可用 表示系统可以正常服务的时间。
      • 可用性 = 平均故障时间 / 系统总运行时间
      • 最基本的要求是：保证3个9或者4个9
    • 高扩展性 表示系统的扩展能力，流量高峰时能否在短时间内完成扩容，更平稳地承接峰值流量。
      • 服务集群、数据库、缓存和消息队列等中间件、负载均衡、带宽、依赖的第三方等

高并发的实践方案有哪些？

  • 纵向扩展 提升单机扩展能力
    • 硬件 加内存，cpu核心数，存储容量，存储速度
    • 软件 减少IO次数，并发或者异步
  • 横向扩展 集群部署以进一步提高并发处理能力
    • 分层架构

具体实践方案

  • 高性能
    • 集群部署，负载均衡
    • 分库分表和索引优化
    • 异步化
    • 限流
      • 前端限流
      • nginx限流
      • 服务端限流
    • 对流量进行削峰填谷，通过MQ承接流量
    • 预计算，比如抢红包场景，可以提前计算好红包金额缓存起来，发红包时直接使用即可。
    • 缓存预热，通过异步任务提前预热数据到本地缓存或者分布式缓存中。
    • 各种池化技术的使用和池大小的设置
    • JVM优化，包括新生代和老年代的大小、GC算法的选择等，尽可能减少GC频率和耗时。
    • 锁选择，读多写少的场景用乐观锁，或者考虑通过分段锁的方式减少锁冲突。
    • 多级别缓存
      • HTTP缓存 包括强制缓存和对比缓存。
      • Content Delivery Network CDN缓存 内容分发网络 CDN 缓存和 HTTP 缓存是好搭档。
      • 负载均衡缓存 相对稳定资源，需要服务协助工作。
      • 进程内缓存 效率高，但容量有限制
      • 分布式缓存 容量大，能力强
    • 缓存场景处理
      • 热点key
      • 缓存雪崩
        • 大量数据同时过期：均匀设置过期时间 二级缓存/双缓存
        • Redis 故障宕机
      • 缓存击穿
        • 某个热点数据过期：互斥锁方案，保证同一时间只有一个业务线程更新缓存 不给热点数据设置过期时间，由后台异步更新缓存
      • 缓存穿透
        • 既不在缓存中，也不在数据库中：布隆过滤器快速判断数据是否存在 非法请求的限制 缓存空值或者默认值
  • 高可用
    • 接口层面的超时设置、重试策略和幂等设计
    • 对等节点的故障转移，Nginx和服务治理框架均支持一个节点失败后访问另一个节点
    • 降级处理：保证核心服务，牺牲非核心服务
    • 限流处理：对超过系统处理能力的请求直接拒绝或者返回错误码
    • MQ场景的消息可靠性保证
  • 高扩展
    • 合理的分层架构
    • 存储层的拆分：按照业务维度做垂直拆分、按照数据特征维度进一步做水平拆分（分库分表）。
    • 业务层的拆分：最常见的是按照业务维度拆（比如电商场景的商品服务、订单服务等）

幂等

  • 定义：幂等指的是相同请求（identical request）执行一次或者多次所带来的副作用（side-effects）是一样的。
  • 实现
    • Token机制
    • 数据库去重表 unique标记
